Hybrid-Cloud
Infrastructure Deployment

Algonquin College  ·  Sep – Nov 2025

Led a team to design and deploy a multi-site hybrid infrastructure simulating a real MSP client environment - virtualized on Proxmox, secured with pfSense, connected over VPN, managed through Active Directory, and protected with automated Veeam backups. Built to work as a coherent system, not a collection of isolated components.

LEAD

Team Leadership & End-to-End Integration

Multiple engineers building separate components simultaneously creates a risk: everything works in isolation but fails when connected. Led the team to prevent exactly that - coordinating parallel workstreams and owning the final integration.

Team LeadTask CoordinationIntegration TestingValidation
Problem
In an MSP simulation with multiple team members each owning different infrastructure layers, the biggest risk isn't whether the firewall works or whether AD works - it's whether they work together. Without someone owning end-to-end integration, components pass individual checks but the system fails at the seams.
Approach
Divided the project into parallel workstreams - virtualization, networking and firewall, identity, and backup - with clear ownership per stream. Set integration checkpoints rather than waiting until the end: each component had to prove it worked with the adjacent layer before moving on. Ran end-to-end validation from each simulated client site after integration - authentication, resource access, inter-site connectivity, and backup execution all verified together.
Outcome
Delivered a fully integrated environment that worked as a system. Parallel workstreams kept the timeline on track, and integration checkpoints caught conflicts early - before they became blockers at final delivery.
FW

Secure Site Connectivity with pfSense

In a real MSP environment, client sites connect over the internet - traffic between them must be encrypted, and access must be explicitly permitted rather than open by default. Getting this wrong means either blocking legitimate traffic or leaving the environment unsecured.

pfSenseFirewall RulesVPN TunnelsPort ForwardingNAT
Problem
The simulated MSP environment had multiple client sites that needed to communicate securely over WAN - replicating the reality that MSP-managed clients are often geographically distributed and connected over public internet links. All inter-site traffic needed to be encrypted, and only explicitly authorized services should be reachable across site boundaries.
Approach
Deployed pfSense instances at each site. Configured firewall rules on a default-deny basis, inter-site traffic blocked unless explicitly allowed. Established encrypted VPN tunnels between sites to secure all cross-WAN communication. Set up port forwarding rules exposing only specific services (RDP, file sharing) to authorized source networks. Tested each rule combination to confirm only intended traffic was passing.
Outcome
All inter-site traffic encrypted and controlled. Verified that services unreachable from unauthorized sites remained blocked while intended connectivity worked correctly; a functioning security boundary, not just network connectivity.
ID

Centralized Identity with Active Directory

Without centralized identity, every server manages its own user accounts independently - which doesn't scale and creates inconsistent access control. AD gives a single place to manage who can access what, enforced consistently across all sites.

Active DirectoryDNSGPORBACDomain Controller
Problem
The multi-site environment needed a single identity authority that all client VMs authenticated against, regardless of which virtual site they were on. Without this, user management would be fragmented across machines and inconsistent access control.
Approach
Promoted a Windows Server VM to domain controller and configured the AD forest. Structured OUs to mirror the simulated client org - users, groups, and workstations organized by function. Applied GPOs to enforce RBAC: users access only what their role permits. Integrated DNS with AD so all domain-joined machines resolved internal resources by name. Validated cross-site authentication by logging in from client VMs at different locations.
Outcome
Centralized identity working across all sites. Cross-site authentication confirmed - users at any client VM could authenticate against the same domain, with access controlled by group membership and enforced by GPO.
DR

Disaster Recovery with Veeam

A backup that has never been restored is not a backup - it's a guess. In a real world, backup is a contractual obligation and a business-continuity requirement. Implementing it means building it, scheduling it, and proving it actually works.

Veeam BackupVM RecoveryRetention PolicyBackup JobsData Protection
Problem
The environment needed verified data protection for its critical VMs - domain controller and file servers especially. An unprotected environment is a liability in any MSP contract. The challenge wasn't just configuring Veeam; it was proving that a restore would actually work when needed.
Approach
Installed Veeam Backup and Replication on a dedicated VM. Configured scheduled backup jobs targeting the domain controller and critical servers first. Set retention policies to balance restore point coverage against storage consumption. Ran test restores against the backup jobs to confirm VMs could be recovered to a known-good state. Documented the backup schedule and recovery procedure as part of the final project deliverables.
Outcome
Verified backup and restore capability across critical VMs - simulated data loss and confirmed recovery. The environment had documented retention policies and a tested recovery process, meeting the standard expected of a real MSP client deployment.
