Windows 11
Enterprise Migration

Ontario Public Service  ·  Jan – Aug 2025

Supported a province-wide upgrade of 65,000+ endpoints from Windows 10 to Windows 11 ahead of the October 2025 EOL deadline. Resolved Autopilot and Intune provisioning failures blocking deployment, fixed post-upgrade compatibility issues affecting day-to-day government operations, and built the knowledge base that let the team scale without constantly re-escalating the same problems.

PROV

Autopilot & Intune Provisioning Failures

Devices not completing enrollment meant they couldn't receive policies, apps, or security configurations - sitting unmanaged or requiring manual IT intervention. At enterprise scale, even a small failure rate creates a significant backlog.

Windows AutopilotMicrosoft IntuneAzure Entra IDDevice EnrollmentProvisioning Logs
Problem
A subset of devices were failing to complete Autopilot enrollment and Intune provisioning - stalling deployment of the Windows 11 migration rollout. Failures were inconsistent across device models and Entra ID groups, with no single obvious cause. Each stuck device either sat unmanaged or required manual remediation, both of which defeated the purpose of a zero-touch deployment model.
Approach
Pulled provisioning logs from affected devices and cross-referenced against Intune device enrollment records in the Azure portal. Grouped failures into repeating categories: Autopilot profile not assigned to the device, Entra ID join failures from missing group membership, Intune policy conflicts blocking enrollment completion, and hardware hash registration issues. Resolved each category systematically. Escalated genuinely novel failures to Tier 3 with full diagnostic context documented, reducing back-and-forth.
Outcome
Provisioning failure categories identified and resolved, keeping the migration rollout moving. Escalation quality improved - Tier 3 received structured diagnostics, cutting resolution time on complex cases.
COMPAT

Post-Upgrade Compatibility Remediation

Windows 11 changed enough security defaults that applications and workflows working fine on Windows 10 silently broke after upgrade - hitting government employees mid-workday, during active public-facing operations.

AuthenticationLOB ApplicationsCredential ManagerSMBKerberos / NTLM
Problem
Post-migration incidents fell into three buckets: authentication failures with government systems, LOB application failures from stricter UAC and runtime changes in Windows 11, and network resource access problems from SMB and credential prompt behaviour differences. Each broke something employees depended on daily - with no warning before the upgrade.
Approach
Triaged incidents by category - authentication, application, or network - before attempting any fix. For auth issues: checked credential manager entries ande device group membership. For application failures: identified whether the cause was a compatibility issue or a policy restriction. For network resource failures: traced to SMB negotiation changes or updated credential prompt behaviour in Windows 11. Documented each resolution pattern as it emerged.
Outcome
Post-upgrade incidents resolved systematically rather than reactively. The categorization approach meant common failure types were handled consistently, and recurring issue types fed directly into the knowledge base - turning one-off fixes into documented solutions.
TRAIN

End-User Training

A technically clean migration that leaves employees unable to find their tools is still a failed migration. Productivity gaps in the days after cutover show up as a flood of "how do I find X" tickets - each one a government employee delayed on public-facing work.

End-User TrainingWindows 11 UIChange ManagementLegacy App Workarounds
Problem
Employees being migrated had no preparation for Windows 11 UI changes - the Start menu, taskbar, and app behaviours all changed. Users with legacy application dependencies had workflows that no longer functioned identically. Without intervention at handoff, the help desk would absorb all of this as ticket volume in the days immediately after migration.
Approach
Delivered targeted 1-on-1 sessions at the point of handoff - covering the highest-impact UI changes for each user's role, not a generic walkthrough. For employees with legacy app dependencies, explained the approved workarounds and set clear expectations on timelines. Adapted each session to the individual rather than following a script. Captured recurring questions from across sessions and fed them back into the knowledge base.
Outcome
Users provided with clear answers to the questions they'd otherwise have called the help desk about. Proactive training at handoff demonstrably reduced the ticket volume in the days following upgrades.
KB

Knowledge Base That Actually Gets Used

Without documentation, every new technician solves the same problem from scratch. The goal wasn't a reference document - it was a troubleshooting guide that a junior analyst or incoming co-op could pick up and act on immediately.

OneNoteSOP WritingKnowledge ManagementHelpdesk Enablement
Problem
The same issues were appearing across multiple analysts and multiple weeks, each person solving them independently from scratch, introducing inconsistency and burning time. There was no shared resource capturing what had already been figured out.
Approach
Built a centralized OneNote troubleshooting guide, explaining how a technician actually encounters a problem. Each entry included the diagnostic steps taken before the fix, not just the fix itself, so the reader understood the reasoning. Entries covered Autopilot/Intune failure types, compatibility incident categories, and common post-migration user issues. It also included resolution for reoccuring issue beyound the scope of migration.
Outcome
Delivered a structured knowledge base used by the help desk and incoming co-op students. Reduced unnecessary escalations to senior staff and gave the team a single consistent source for troubleshooting - turning individual problem-solving into shared institutional knowledge.
