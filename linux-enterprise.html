NET

Server Network Configuration

Configured the Linux server's network interfaces from the ground up - static IPs, routing, and hostname resolution - forming the stable foundation every other service depends on.

NetworkManagerStatic IPRoutingDNS Resolver
Problem
Every service on the server - DNS, mail, LDAP, web - relies on the network layer being predictable. A server with a dynamically assigned address can change IPs unexpectedly, breaking DNS records and service configurations across the environment.
Approach
Configured two network interfaces - one bridged to the internal network with a static IP, one NAT for internet access. Set static IP parameters, gateway, and DNS resolver settings through interface configuration files. Verified routing tables and confirmed name resolution before moving on to any other service.
Outcome
Stable, predictable network foundation in place. All subsequent services configured against fixed addresses that wouldn't change between reboots, eliminating a whole category of intermittent failures.
DIAG

Network Diagnostics & Firewall Control

Applied the core Linux network toolset to verify services, test connectivity, capture live traffic, and enforce firewall rules - the same tools used in production to diagnose issues and control access.

tcpdumpnetcatiptablesConnection MonitoringPacket Analysis
Problem
When a service isn't reachable, the cause could be a firewall rule blocking traffic, a service not listening on the expected port, or a routing problem. Each requires a different tool to diagnose - and guessing wastes time in any real environment.
Approach
Used connection monitoring tools to confirm which services were actively listening and on which interfaces. Used netcat to test raw TCP connectivity between server and client independent of any application. Wrote iptables rules to allow traffic from authorized networks while dropping all other sources on specific ports. Captured and filtered live traffic with tcpdump to confirm packets were arriving at the right interface.
Outcome
Each service's network behaviour verified independently before layering on the next. Firewall rules in place and confirmed working - the environment was locked down by default, with access explicitly permitted only where needed.
SSH

Secure Remote Access with SSH & Public-Key Authentication

Hardened remote access by replacing password-based SSH with public-key authentication - eliminating brute-force as an attack vector - and restricting access to authorized source networks at the firewall level.

OpenSSHRSA Key PairsPublic-Key Authiptables
Problem
Password-based SSH is a constant target - automated bots continuously attempt common credentials against any exposed port 22. A single weak password on any account is enough for a breach. The default configuration is not secure enough for any server that matters.
Approach
Generated RSA key pairs on the client and deployed public keys to each user's authorized keys on the server, including root. Updated the SSH server configuration to accept only key-based authentication, disabling password login entirely. Applied firewall rules restricting SSH access to the designated client network only.
Outcome
Passwordless login confirmed working for all required accounts. Brute-force attack surface eliminated. Login events verified in the system auth log, remote access fully auditable and restricted to authorized users from authorized networks.
DNS

Authoritative DNS with Primary / Secondary Replication

Deployed a full BIND DNS environment - a primary authoritative name server for a custom domain with forward and reverse zones, and a secondary server that automatically replicates zone data for redundancy.

BINDAuthoritative DNSZone ReplicationForward / Reverse ZonesRedundancy
Problem
Every service in the environment depends on DNS to find other services by name. A single DNS server is a single point of failure, if it goes down, nothing resolves and the entire environment effectively stops functioning, even if all other services are still running.
Approach
Installed BIND and wrote zone files for the forward zone covering all server hostnames, and a reverse zone mapping IPs back to hostnames - required for mail authentication and system logging. Configured the primary server as authoritative. Deployed a secondary server and configured automatic zone transfers so it stays current whenever records change, with no manual sync required.
Outcome
Both primary and secondary DNS servers answering queries correctly. Zone replication confirmed working, a change on the primary propagated to the secondary automatically. Reverse lookups verified for all hosts, enabling correct mail delivery and audit log resolution.
WEB

Web Server with Apache & SSL/TLS

Deployed Apache hosting multiple sites on a single server using virtual hosting - with sites secured by SSL/TLS. The same architecture used by the majority of Linux-based web environments in production.

ApacheVirtual HostingSSL / TLSHTTPS
Problem
Hosting one site per server is wasteful and doesn't reflect how production environments work. Multiple sites need to coexist on shared infrastructure, cleanly separated by domain - and needs to be served securely over HTTPS, which is now the baseline expectation for any web-facing service.
Approach
Configured Apache with a custom configuration file defining three virtual hosts serving separate domains from their own document roots, with SSL enabled, pointing to the certificate and key files. Each site had independent logging.
Outcome
All three sites independently reachable and serving the correct content. HTTPS site confirmed secure with no certificate errors. Access and error logs clean - the server handling multiple sites correctly with no configuration bleed between them.
FILE

Cross-Platform File Sharing with Samba & NFS

Configured two complementary file sharing protocols - Samba for Windows-Linux access over SMB, and NFS for Linux-to-Linux network storage - each with appropriate access controls for different client types.

SambaNFSSMB ProtocolAccess ControlNetwork Mounts
Problem
Most enterprise environments are mixed - Linux servers serving Windows workstations and other Linux systems. A single file sharing solution doesn't cover both. Samba speaks SMB natively to Windows clients; NFS is the standard for Linux-to-Linux storage. Running both means every client type gets the right protocol without compromise.
Approach
Configured Samba with user accounts and shares covering home directories and custom access paths. Validated the configuration before restarting the service. Tested access from both a Linux client and a Windows host mapping a network drive. Configured NFS exports with differentiated permission tiers - read-only for some shares, read-write for others - and mounted them on the Linux client to confirm enforcement.
Outcome
Windows clients accessing Linux shares over SMB with no compatibility issues. NFS mounts working on Linux clients with permission tiers enforced as configured. Both protocols running concurrently on the same server - each serving its intended client type correctly.
MAIL

Mail Server with Postfix

Deployed Postfix as the mail transfer agent - configuring inbound and outbound delivery, monitoring the mail queue, and diagnosing delivery failures using delivery status codes.

PostfixSMTPMail QueueDelivery Diagnostics
Problem
Email delivery is opaque by default - a message either arrives or it doesn't, with no obvious indication of why. In any environment relying on mail for alerts, reports, or user communication, the ability to monitor the queue and read delivery status is the difference between knowing something failed and finding out hours later.
Approach
Configured Postfix with domain and mailbox settings and managed the service through systemctl. Sent and received email between server and client users, monitoring delivery in real time through the mail log. Inspected the deferred queue to identify held messages, diagnosed failures using delivery status codes, and cleared the queue after resolving the cause.
Outcome
Mail flowing reliably between users in both directions. Queue monitoring and delivery diagnostics confirmed working - able to identify, diagnose, and resolve delivery failures rather than simply observing that mail wasn't arriving.
DIR

Centralized Directory Services with OpenLDAP

Deployed an LDAP directory server as a single source of truth for user accounts and host records - so any client on the network can authenticate and resolve resources centrally, instead of managing accounts independently on every machine.

OpenLDAPDirectory ServicesCentralized AuthUnix Account Objectsiptables
Problem
Without a central directory, user accounts exist independently on every server. Adding a user means creating accounts on every machine they need access to; removing one means finding and deleting accounts everywhere. At any real scale, that's unmanageable - and it means there's no single authoritative record of who has access to what.
Approach
Installed and configured OpenLDAP with the appropriate schemas for Unix user accounts and host objects. Populated the directory with user entries and host records using LDIF files. Verified the directory was queryable from both server and client. Configured the client to resolve host lookups against LDAP. Applied a firewall rule restricting directory access to authorized networks only.
Outcome
Centralized directory operational - user accounts and host records stored in one place and accessible from any authorized client on the network. Host lookups resolving via LDAP confirmed. Directory access restricted at the firewall - not exposed beyond the intended network boundary.
