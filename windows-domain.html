GPO

Centralized Software Deployment via Group Policy

Built a GPO-based software distribution pipeline - giving IT the ability to push, upgrade, and retire applications across the entire organization based on a user's department, with no manual intervention on end-user machines.

Group PolicySoftware DeploymentActive Directory OUsMSI Packaging
Problem
Managing software manually across dozens of machines leads to version inconsistencies, missed updates, and security gaps. IT needs a way to deploy, update, and retire applications organization-wide from a single point, without touching individual endpoints and without depending on users to install things themselves.
Approach
Created GPOs scoped to specific OUs in Active Directory. Used Assigned deployment for mandatory software - silently auto-installs on login with no user prompt. Used Published for optional tools - user self-serves via Control Panel. Managed separate software bundles per department. Ran a live upgrade replacing a legacy package with its successor using GPO's upgrade policy, and retired software gracefully - removing the deployment while preserving already-installed copies for existing users.
Outcome
Software deployment, upgrade, and retirement all controlled from a single GPO console. The right tools automatically present for each department's users on login - no manual installs, no version drift, no IT intervention needed per-machine.
DNS

Internal DNS Infrastructure

Active Directory can't function without DNS. Exchange can't route mail without reverse DNS. Built the domain's authoritative DNS layer - forward and reverse zones, full record coverage - as the foundation everything else depended on.

Windows DNSForward ZonesReverse ZonesPTR RecordsSOA / NS
Problem
AD relies entirely on DNS to locate domain controllers - without it, machines can't join the domain and users can't authenticate. Exchange requires reverse DNS to pass mail authentication. DNS isn't a nice-to-have; it's the foundational service that determines whether the rest of the environment functions at all.
Approach
Configured forward lookup zones in Windows DNS Manager with A records for every server on the network. Built a reverse lookup zone under the correct namespace and created PTR records for each host so reverse resolution worked across all machines. Verified zone authority by checking SOA and NS records - confirming the server was the designated authoritative source for the domain.
Outcome
Full forward and reverse name resolution working across the domain. AD authentication, Exchange mail routing, and all inter-service communication functioned correctly - DNS doing exactly what it needs to do: staying invisible while everything else works.
MAIL

Enterprise Email with Microsoft Exchange

Deployed and administered a full Exchange Server environment - mailbox quotas, outbound SMTP routing, cross-domain mail flow, compliance retention, and full client configuration for both Outlook and Thunderbird.

Exchange ServerSMTPMailbox ManagementPowerShellRetention Policies
Problem
Enterprise email requires active management: storage quotas prevent one user from filling shared disk, outbound routing needs to be controlled for deliverability and inspection, and retention policies are a compliance requirement - organizations cannot allow employees to delete relevant email during legal proceedings or audits. All of this needs to be configured, not just installed.
Approach
Configured per-mailbox storage quotas (warning, send limit, receive cutoff) in the Exchange Admin Center. Created an SMTP Send Connector routing all outbound mail through a smart host. Applied a retention hold to suspend policy enforcement on a specific mailbox - simulating an employee leave scenario. Configured Outlook and Thunderbird clients and verified cross-domain mail flow between domains.
Outcome
Fully operational Exchange environment with enforced storage governance, controlled outbound routing, compliance-ready retention hold capability, and verified email delivery between domains and between Outlook and Thunderbird clients.
VM

Cloud Server Infrastructure on Microsoft Azure

Deployed a Windows Server VM in Azure with proper network segmentation, availability zone placement, and secure remote access - without exposing a public RDP port that attackers actively scan for.

Azure VMAzure BastionVirtual NetworkAvailability ZonesResource Groups
Problem
Standing up a cloud VM securely requires more than clicking "deploy" - a publicly exposed RDP port is a constant target for brute-force attacks. Availability and cost management require intentional choices around zone placement and resource grouping from the start.
Approach
Provisioned a Standard B1ms VM in a dedicated resource group inside a virtual network with a dedicated subnet. Placed the VM in an availability zone for resilience. Enabled Azure Bastion for browser-based RDP - no public IP on the VM, no port 3389 exposed to the internet. Managed the full VM lifecycle and monitored health events through the Azure activity log.
Outcome
A running cloud VM accessible only through Bastion, removing the RDP attack surface entirely. Structured resource group setup gave clean billing separation and lifecycle management, matching how teams manage Azure resources in production.
STORE

Cloud Object Storage with Azure Blob

Deployed a private Azure Blob storage environment with full diagnostic logging - ensuring data access is locked down by default and every operation is captured for security auditing.

Azure Blob StorageAccess ControlDiagnostic LoggingStorage Browser
Problem
Misconfigured public access on Azure storage containers has been the root cause of major data breaches. The default should always be private - and beyond access control, security teams need a record of who accessed what data and when. Neither happens without explicit configuration.
Approach
Created a Storage Account and provisioned Blob containers with anonymous access disabled at both the container and account level. Enabled storage diagnostics routing all read, write, and delete operations to a dedicated log container - creating a tamper-evident audit trail. Used Azure Storage Browser to manage contents and verified log capture from the portal.
Outcome
Private-by-default storage with full operational logging. Every access event captured, answering the key security audit question of who accessed what, and when.
IAM

Cloud Identity & Access Management with Azure RBAC

Implemented least-privilege access control across Azure resources - users get only the permissions their role requires, with every change logged and auditable in Activity Logs.

Azure RBACIAMLeast PrivilegeActivity LogsAccess Control (IAM)
Problem
Broad permissions granted for convenience are how breaches and accidental deletions happen. RBAC enforces least privilege: a developer can deploy but can't touch billing; a support tech can read logs but can't delete infrastructure. It's a baseline control required by ISO 27001, SOC 2, and NIST - and it needs to be verified, not just assigned.
Approach
Assigned Azure built-in roles at resource group scope via Access Control (IAM) - scoped to the minimum necessary. Used the "Check access" feature to verify effective permissions before and after assignment. Validated enforcement by logging into Azure with a restricted account and confirming access boundaries.
Outcome
Least-privilege access enforced and verified. A complete timestamped audit trail of every access modification, demonstrating the kind of IAM hygiene cloud security audits look for.
